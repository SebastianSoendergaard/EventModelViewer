# Event Model Viewer - Architecture Guide

## Overview

Event Model Viewer is built as a **modular single-page application** that compiles from source files into a self-contained HTML file. This document explains the architectural patterns and design decisions.

## Core Architectural Patterns

### 1. EventBus Pattern (Pub/Sub)

**The EventBus is the backbone of module communication.** Modules never call each other directly - all communication goes through the EventBus.

**Location:** `src/event-bus/event-bus.js`

**Key Design Decision:** EventBus is **NOT wrapped in an IIFE** (see `build.js:9`) so it's globally accessible to all modules.

#### Event Catalog

| Event | Payload | Purpose | Emitted By |
|-------|---------|---------|------------|
| `APP_INIT` | `{}` | Application initialized | `app.html` |
| `FILE_LOADED` | `{ json, fileName }` | JSON file loaded | `file-buttons` |
| `JSON_CHANGED` | `{ json, source }` | JSON modified | `code-view`, `tree-view`, `editor` |
| `FILTER_TOGGLED` | `{ type, checked }` | View filter changed | `filter-toggles` |
| `EDITOR_RESIZED` | `{}` | Panel resized | `resizer` |
| `TREE_SYNC` | `{}` | Request tree view to re-render | `editor` (on tab switch) |
| `CODE_SYNC` | `{ json }` | Request code editor to sync | `editor` (on tab switch) |

#### Source-Aware Pattern

The `source` field in `JSON_CHANGED` prevents infinite update loops:

```javascript
// Emitter
EventBus.emit(Events.JSON_CHANGED, { 
    json: newJson, 
    source: 'tree'  // Identifies who made the change
});

// Subscriber
EventBus.on(Events.JSON_CHANGED, ({ json, source }) => {
    if (source === 'tree') return;  // Skip own updates
    updateTreeView(json);
});
```

**Source values:**
- `'code'` - From code editor
- `'tree'` - From tree view
- `'history'` - From undo/redo (special: doesn't push to history stack)
- `'addslice'` - From add slice button

### 2. Build System

**Custom Node.js bundler** that assembles modular source into a single HTML file.

**Location:** `build.js` (root)

#### Build Process

1. **Read source files** - Scans `src/` for HTML/CSS/JS triplets
2. **IIFE wrapping** - Wraps each module's JS in an IIFE for scope isolation
3. **Assembly** - Injects modules into placeholder comments in `app.html`
4. **Output** - Writes `event-model-viewer.html`

#### Module Order Matters

The build script defines module order (see `build.js:17-54`):
- EventBus **first** (must be global)
- Core modules (toolbar, resizer)
- Editor modules
- Viewer modules (diagram before zoom-export)

**Why order matters:** Some modules reference functions from others. The build order ensures dependencies are available.

#### Placeholder System

`app.html` contains comment tokens that `build.js` replaces:

**CSS tokens:**
```html
/* TOOLBAR_CSS */
/* RESIZER_CSS */
/* EDITOR_CSS */
/* VIEWER_CSS */
```

**JavaScript tokens:**
```javascript
// TOOLBAR_JS
// RESIZER_JS
// EDITOR_JS
// VIEWER_JS
```

The build script (`build.js:84-89`) finds these exact tokens and injects the corresponding module code.

### 3. State Management

**Distributed local state** - No single source of truth. Each module maintains its own copy of the JSON data.

#### State Flow

```
User Action
    ‚Üì
Module A modifies local JSON
    ‚Üì
EventBus.emit(JSON_CHANGED, { json, source: 'A' })
    ‚Üì
Module B receives event
    ‚Üì
if (source !== 'B') { updateLocalCopy(json); }
```

#### State Persistence

**localStorage** automatically saves:

| Key | Value | Updated On | Module |
|-----|-------|------------|--------|
| `eventModelJson` | JSON string | Every `JSON_CHANGED` | `file-buttons` |
| `eventModelFileName` | Filename | File upload | `file-buttons` |
| `eventModelViewerLayout` | Panel widths | Resize | `resizer` |

**Auto-restore:** On page load, `file-buttons` checks localStorage and emits `FILE_LOADED` if data exists.

### 4. IIFE Encapsulation

Most modules are wrapped in **Immediately Invoked Function Expressions** for scope isolation:

```javascript
(function() {
    // Module-private variables
    let _jsonData = null;
    
    // Module initialization
    EventBus.on(Events.APP_INIT, () => {
        initialize();
    });
    
    // Private functions
    function initialize() {
        // Setup code
    }
})();
```

**Exception:** EventBus is NOT wrapped so it's globally accessible.

**Result:** Modules can't access each other's internals - they must communicate via EventBus.

### 5. Code Editor Integration

**Technology:** ACE Editor (not CodeMirror, despite variable name)

**Why ACE?** See `docs/code-editor-selection-learnings.md` - CodeMirror 6 attempt failed due to:
- Complex module system incompatible with single-file build
- Tree-shaking issues
- CDN loading problems

#### Variable Naming Legacy

The global variable is named `codeMirrorView` but uses **ACE Editor**:

```javascript
// In code-view.js
window.codeMirrorView = ace.edit('code-editor');
```

**Why the confusing name?** Historical artifact from the failed CodeMirror attempt. Kept for backward compatibility with existing code references.

**Actual API:**
- `codeMirrorView.setValue(text, -1)` - ACE method
- `codeMirrorView.getValue()` - ACE method
- `codeMirrorView.session.setMode('ace/mode/json')` - ACE API

**Custom HTML/CSS Grid + SVG** (not Mermaid.js or canvas)

**Location:** `src/viewer/diagram/diagram.js`

**Key Functions:**
- `generateEventModelDiagram()` - Main entry point (line 84)
- `drawArrow()` - SVG path rendering (line 383)
- `drawAllArrows()` - Batch arrow rendering (line 186)
- `generateTrigger/Command/Event/View/Tests()` - Element generators

#### Why Custom Rendering?

Event Model diagrams require:
- **Precise grid layout** (swimlanes √ó slices)
- **Dynamic arrow routing** (avoiding element collisions)
- **Interactive elements** (collapse, filters)

Mermaid.js couldn't handle this complexity, so we built custom rendering.

### 7. View Synchronization

**Three views** must stay synchronized:
1. **Code View** (ACE Editor)
2. **Tree View** (drag-drop tree)
3. **Diagram** (visualization)

#### Sync Strategy

**Tab-switch sync:**
```javascript
// When switching to Code tab
EventBus.emit(Events.CODE_SYNC, { json: currentJson });

// When switching to Tree tab
EventBus.emit(Events.TREE_SYNC, { json: currentJson });
```

**Continuous sync:**
- Diagram always updates on `JSON_CHANGED` (no source filter)
- Code/Tree skip updates from their own source

**History integration:**
- Undo/redo uses `source: 'history'` to avoid double-pushing to history stack

### 8. History Management

**Unified undo/redo** across both editors.

**Location:** `src/editor/editor.js`

#### History Stack

The history manager uses a **single array with index-based navigation** (not separate undo/redo stacks):

```javascript
historyManager = {
    states: [],         // All historical states (max 50)
    currentIndex: -1,   // Current position in states array
    maxStates: 50,      // Circular buffer limit
    isRestoring: false  // Flag to prevent double-pushing during undo/redo
}
```

**How it works:**
- `pushState()` - Adds new state, truncates future states if in middle of history
- `undo()` - Decrements `currentIndex`, restores `states[currentIndex]`
- `redo()` - Increments `currentIndex`, restores `states[currentIndex]`
- `isRestoring` flag prevents `pushState()` from being called during restore

#### Debouncing

- **Code editor:** 1000ms debounce (don't push every keystroke, also prevents excessive diagram re-renders)
- **Tree editor:** Immediate (each action is atomic: drag-drop, inline edit, context menu)

**Why different?** Code edits are incremental (typing); tree actions are discrete user operations.

## Error Handling

**Pattern:** Try/catch at I/O boundaries, console logging only.

**No centralized error handling** - each module handles its own errors.

**Key boundaries:**
- localStorage read/write (quota, corruption)
- JSON.parse (file upload, editor changes)
- Diagram rendering (invalid JSON structure)
- SVG export (canvas conversion)

**Recovery strategies:**
- localStorage errors ‚Üí `removeItem()` and retry
- JSON parse errors ‚Üí Show error message, keep last valid state
- Rendering errors ‚Üí Show error div, don't crash app

## Performance Considerations

### Debouncing

**Code editor updates:** 1000ms debounce prevents excessive diagram re-renders during typing.

### Lazy Rendering

**Tree view:** Only renders visible nodes when collapsed (performance optimization for large JSON).

### Arrow Calculation Caching

**Diagram:** Caches arrow paths to avoid recalculation on every render (future enhancement - not yet implemented).

## Security Considerations

**localStorage only** - No server communication for JSON data.

**XSS Protection:** All user JSON values are escaped via the `escapeHtml()` function before being rendered into the diagram's innerHTML (`diagram.js`). Raw values are never directly inserted into DOM.

**File Access:** FileReader API (browser-sandboxed).

## Deployment Model

**Self-contained HTML file** (with CDN dependencies) that can be:
- Checked into git with event model JSON files
- Served from any static file server
- Opened directly from filesystem (file://)
- Emailed or shared

**Runtime dependencies** (loaded from CDN):
- ACE Editor (always loaded)
- html2canvas (loaded only during PNG export)

**Why this model?** Event Model diagrams should live with the code. A single-file viewer ensures version compatibility - no "viewer too old for diagram format" issues.

## Future Architecture Considerations

### Potential Improvements

1. **Module bundler migration** - Replace custom build.js with Rollup/Vite
2. **TypeScript** - Add type safety to EventBus contracts
3. **Web Components** - Encapsulate modules as custom elements
4. **Virtual scrolling** - Handle huge JSON files (1000+ slices)
5. **Service Worker** - Offline-first with caching

### Architectural Constraints

**Must maintain:**
- Single-file output (core value prop)
- No build-time dependencies (just Node.js + build.js)
- Backward compatibility with existing JSON format

## Troubleshooting Common Issues

### "Function X is not defined"

**Cause:** Module loaded out of order or IIFE wrapped when it shouldn't be.

**Fix:** Check `build.js` module order or unwrap IIFE if function needs to be global.

### "EventBus is not defined"

**Cause:** EventBus not loaded first, or IIFE-wrapped by mistake.

**Fix:** Ensure EventBus is listed first in `build.js` and NOT wrapped in IIFE.

### Changes to src/ not appearing

**Cause:** Forgot to rebuild.

**Fix:** Run `node build.js` after every source change.

### localStorage quota exceeded

**Cause:** Very large JSON files (>5MB on some browsers).

**Fix:** Code includes error handling (`removeItem` + retry) but may need to add size warning.
